# 04. 순서도 → 코드 역방향 변환 + 양방향 동기화

## 사용한 에이전트 도구

- Claude Code (CLI)

## 에이전트에게 위임한 작업의 범위

### 작업 범위

- `flow-to-code.ts`: React Flow 노드/엣지 그래프 → 달빛약속 코드 생성 알고리즘
  - `generateBlock` 재귀 순회, visited + stopIds로 무한루프 방지
  - decision(조건문) BFS merge 탐색, loop body tail 역추적
- `NodePalette.tsx`: 8종 노드 드래그 팔레트
- `NodeEditModal.tsx`: 노드 타입별 입력 폼 + 실시간 유효성 검사 + 코드 미리보기
- 양방향 동기화 (`sync.ts` 전략 설계):
  - `lastEditSource`: 마지막 편집 방향 추적 (`'code'` | `'flowchart'`)
  - `isProgrammaticUpdateRef`: F2C가 Monaco setValue 시 onChange 무시 처리
  - 코드→순서도: 300ms debounce + `lastEditSource !== 'flowchart'` 조건
  - 순서도→코드: 즉시 실행 + programmatic flag

### 판단 근거

- F2C 알고리즘은 그래프 순회 로직이 복잡하므로, 구현 전 에이전트와 전략을 충분히 논의한 후 위임
- 동기화 루프 방지는 구현 패턴이 명확하므로 에이전트가 작성하고 직접 시나리오 테스트로 검증
- NodeEditModal의 12개 필드는 구조가 반복적이므로 에이전트가 `useReducer` 기반으로 일괄 처리

## 에이전트의 산출물을 검증한 방법

- 순서도에서 노드를 연결하고 생성된 코드가 달빛약속 문법에 맞는지 직접 확인
- 생성된 코드를 실행하여 의도한 결과가 나오는지 확인
- 코드→순서도→코드 왕복 후 동등성 확인 (roundtrip 테스트)
- 양방향 편집 시나리오 직접 테스트: 코드 편집 → 순서도 반영 → 순서도 수정 → 코드 반영, 무한루프 없음 확인

## 에이전트가 잘못된 방향을 제시했을 때 교정한 방법

### 교정 1: 반복문 body 내용 중복 출력

F2C에서 반복문 본문 내용이 두 번 출력되는 버그.

- **원인 분석**: `bodyEdge` 선택 로직이 body 진입 엣지와 exit 엣지를 구분하지 못해 body를 두 번 순회
- **교정**: `edgeType: 'body'` 태그를 도입하여 body 진입 엣지를 명시적으로 구분. `nextEdge`는 `'body'` edgeType 제외, `bodyEdge`는 `'body'` 태그 우선 선택

### 교정 2: ESLint `isProgrammaticUpdate` 불변성 오류

에이전트가 `isProgrammaticUpdate`를 일반 변수로 선언했으나 ESLint가 ref로 인식하지 못해 오류 발생.

- **교정**: `isProgrammaticUpdateRef`로 이름 변경하여 ref임을 명시

### 교정 3: NodeEditModal 12개 useState → useReducer

에이전트가 처음에 12개의 개별 `useState`를 사용하여 ESLint `set-state-in-effect` 경고 발생.

- **교정**: `FormState` 타입 + `formReducer` + `FORM_INIT`으로 `useReducer` 일괄 관리로 교체 지시

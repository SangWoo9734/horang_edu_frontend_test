# 프로젝트 회고

---

## 잘된 점

### 사전 문서화가 에이전트 협업의 품질을 결정했습니다

구현에 들어가기 전에 요구사항 명세, 상세 기획서, 스킬 문서 등을 꽤 공들여 작성했습니다. 처음에는 "이걸 다 쓰고 시작하면 시간이 부족하지 않을까" 싶었는데, 결과적으로 에이전트가 일관된 방향으로 구현을 진행할 수 있었던 건 이 문서들 덕분이었습니다. AST → 순서도 노드 매핑 규칙이나 양방향 동기화 전략 같은 것을 미리 정의해두니까, 에이전트에게 "이거 만들어줘"라고 했을 때 의도에서 크게 벗어나는 경우가 적었습니다. 가이드와 스펙을 충분히 준비하는 것이 에이전트 활용의 안정성에 직접적인 영향을 준다는 것을 실감한 경험이었습니다.

### 검증 루프가 완성도를 만들었습니다

구현 대부분을 에이전트가 위임받아 진행했고, 핵심 기능도 달빛약속에서 제공하는 API를 활용했기 때문에, "내가 직접 잘 짰다"는 식의 성취감을 느끼기는 어려웠습니다. 대신 에이전트의 산출물을 지속적으로 검증하는 과정이 결과물의 품질을 좌우했다고 생각합니다. 가장 까다로웠던 노드 포커싱 문제도, 여러 대안을 비교하고 각각을 실제로 테스트해본 뒤 최선의 방식을 선택하는 과정을 거쳐 안정적으로 해결할 수 있었습니다.

## 아쉬웠던 점

### 아이디에이션에 예상보다 많은 시간을 썼습니다

초기에는 초·중등 학습자를 타겟으로 달빛약속을 활용한 새로운 학습 제품을 만들고 싶었습니다. 코드 퍼즐 게임, 듀오링고 스타일 학습 플랫폼 등 여러 방향을 구체적으로 기획해봤는데, 기획을 구체화할수록 호랑에듀의 기존 서비스와 방향이 겹친다는 것을 알게 됐습니다. 호랑에듀 실제 서비스를 탐색하면서 단계별 학습, 문제 풀이, 알고리즘 문제까지 이미 커버하고 있다는 것을 확인했고, 결국 과제 예시 중 하나를 발전시키는 방향으로 전환했습니다.

돌이켜보면 이 과정이 아깝기만 한 것은 아니었습니다. 여러 방향을 탐색하면서 "달빛약속의 어떤 기능이 기존에 없는 가치를 만드는가"를 깊게 생각할 수 있었고, 최종 선택인 코드 ↔ 순서도 시각화 도구가 왜 적합한지에 대한 근거도 명확해졌습니다. 하지만 결과적으로 구현에 온전히 집중할 시간이 줄어든 것은 사실이고, 아이디에이션 단계에서 "빠르게 결정하고 실행에 집중하는" 판단이 더 필요했습니다.

### 실행 노드 하이라이트의 기술적 한계

마지막 실행 노드의 하이라이트 처리가 기술적으로 가장 아쉬운 부분입니다. 여러 대안을 비교 검토하고 선택한 결과이지만, 브라우저 렌더링 타이밍에 의존하는 방식이라 완전히 매끄럽지 않은 순간이 있습니다. 좀 더 시간이 있었다면 React Flow의 내부 상태와 동기화하는 방식을 더 깊게 파볼 수 있었을 것 같습니다.

## 시간이 더 있었다면

### 학습 경험으로의 확장

지금은 자유롭게 코드를 작성하고 순서도를 확인하는 형태인데, 호랑에듀의 기존 서비스처럼 문제와 함께 순서도를 제공하는 형태로 발전시킬 수 있었을 것 같습니다. 예를 들어 "이 순서도를 보고 코드를 작성해보세요" 같은 문제를 제공하면, 순서도와 코드의 관계를 더 능동적으로 학습하는 경험이 될 수 있습니다. 달빛약속이라는 언어가 가진 교육적 특성을 살려 새로운 학습 경험을 만드는 방향을 다시 시도해보고 싶습니다.

### 디버거로서의 완성도

브레이크포인트 설정, 오류 발생 지점을 순서도에서 직관적으로 표시하는 기능, 실행 이력을 타임라인으로 탐색하는 기능 등 디버거로서의 기능을 보강하면 도구의 실용성이 크게 올라갈 것입니다.

### 에이전트 활용 고도화

이번에는 Claude Code 하나로 구현 전반을 진행했는데, 모델별로 역할을 나눠보지 못한 것이 아쉽습니다. 문서 작성과 기획에는 GPT 계열을, 코드 구현에는 Claude Code를 활용하는 식으로 각 모델의 특성에 맞게 분배했다면, 더 효율적인 협업이 가능했을 것 같습니다.
